## 3. 인시던트 대응 플레이북

### 3.1 DB 장애 (PostgreSQL Down)

**증상**: API 5xx 급증, "Connection refused" 로그

**즉시 조치 (5분 이내)**:
1. `kubectl get pods -n production -l app=postgresql` 상태 확인
2. RDS 콘솔에서 DB 인스턴스 상태 확인
3. 읽기 전용 모드 가능 여부 확인 (Read Replica)

**복구 절차**:
```bash
# RDS 상태 확인
aws rds describe-db-instances --db-instance-identifier synapse-prod

# Read Replica로 페일오버 (Multi-AZ)
aws rds failover-db-cluster --db-cluster-identifier synapse-prod-cluster

# 커넥션 풀 리셋
kubectl rollout restart deployment/note-service -n production
kubectl rollout restart deployment/card-service -n production
```

**후속 조치**:
- RCA(Root Cause Analysis) 작성
- 모니터링 임계값 재검토
- 필요 시 DB 스펙 업그레이드

### 3.2 LLM 비용 급증

**증상**: LangSmith 비용 대시보드 $100/일 초과 알림

**즉시 조치**:
1. LangSmith에서 비정상 호출 패턴 확인
2. 특정 사용자/IP의 과도한 호출 여부 확인
3. Rate Limiter 긴급 적용

**복구 절차**:
```bash
# AI 서비스 Rate Limit 긴급 조정
kubectl set env deployment/ai-service \
  RATE_LIMIT_PER_USER=5 \
  RATE_LIMIT_WINDOW=3600 \
  -n production

# 비정상 사용자 차단 (필요 시)
curl -X POST https://api.synapse.app/admin/users/{userId}/suspend \
  -H "Authorization: Bearer $ADMIN_TOKEN"
```

**근본 대책**:
- 사용자별 일일 토큰 한도 설정
- 비용 알림 임계값 세분화
- 캐싱 레이어 강화 (유사 요청 재사용)

### 3.3 테넌트 격리 위반

**증상**: 사용자가 다른 사용자의 데이터를 조회함

**즉시 조치 (P0)**:
1. 해당 API 엔드포인트 즉시 차단
2. 영향 범위 파악 (접근 로그 분석)
3. 영향받은 사용자 식별

**복구 절차**:
```bash
# 긴급: 문제 엔드포인트 차단 (Gateway Level)
kubectl apply -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: emergency-block
  namespace: production
spec:
  # 해당 서비스 격리
EOF

# RLS 정책 확인
psql -h $DB_HOST -U $DB_USER -d synapse -c "
  SELECT * FROM pg_policies WHERE tablename='notes';
"

# 감사 로그 추출
psql -h $DB_HOST -U $DB_USER -d synapse -c "
  SELECT * FROM audit_log 
  WHERE event_type='DATA_ACCESS' 
  AND timestamp > now() - interval '1 hour'
  ORDER BY timestamp DESC;
"
```

**후속 조치**:
- 보안 사고 보고서 작성
- 영향받은 사용자 개별 통지
- RLS 정책 전수 검사
- 추가 통합 테스트 작성

### 3.4 Stripe Webhook 실패

**증상**: 결제 완료 후 플랜 미반영, Stripe 대시보드 Webhook 실패 표시

**즉시 조치**:
1. Stripe Dashboard > Webhooks > Failed Events 확인
2. 서비스 로그에서 Webhook 처리 에러 확인
3. 수동으로 미처리 이벤트 재처리

**복구 절차**:
```bash
# Webhook 엔드포인트 상태 확인
curl -s https://api.synapse.app/webhooks/stripe/health

# 미처리 이벤트 재시도 (Stripe CLI)
stripe events resend evt_xxxxx --webhook-endpoint we_xxxxx

# 수동 동기화 (관리자 API)
curl -X POST https://api.synapse.app/admin/billing/sync \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"userId": "usr_xxxxx"}'
```

### 3.5 Redis 장애/페일오버

**증상**: 응답 지연 급증, 캐시 미스율 100%

**즉시 조치**:
1. Redis Cluster 상태 확인
2. 페일오버 자동 진행 여부 확인
3. 애플리케이션의 캐시 폴백(DB 직접 조회) 동작 확인

**복구 절차**:
```bash
# Redis Cluster 상태
redis-cli -h $REDIS_HOST -p 6379 cluster info
redis-cli -h $REDIS_HOST -p 6379 cluster nodes

# 수동 페일오버 (필요 시)
redis-cli -h $REDIS_REPLICA -p 6379 cluster failover

# 앱 캐시 워밍업
curl -X POST https://api.synapse.app/admin/cache/warmup \
  -H "Authorization: Bearer $ADMIN_TOKEN"
```

## 5. 백업/복구

### 5.1 백업 정책

| 대상 | 방식 | 주기 | 보관 기간 |
|------|------|------|-----------|
| PostgreSQL | RDS 자동 스냅샷 | 매일 | 30일 |
| PostgreSQL | 수동 스냅샷 | 배포 전 | 90일 |
| Elasticsearch | Snapshot to S3 | 매일 | 14일 |
| Redis | RDB + AOF | 1시간 | 7일 |
| S3 (첨부파일) | Cross-Region Replication | 실시간 | 무기한 |

### 5.2 복구 절차 (PostgreSQL)

```bash
# 1. 복구 대상 스냅샷 확인
aws rds describe-db-snapshots --db-instance-identifier synapse-prod \
  --query 'DBSnapshots[*].[DBSnapshotIdentifier,SnapshotCreateTime]'

# 2. 스냅샷에서 새 인스턴스 생성
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier synapse-prod-restored \
  --db-snapshot-identifier rds:synapse-prod-2026-05-07-00-00

# 3. 데이터 검증 후 DNS 전환
# (주의: 다운타임 발생, 사전 공지 필요)

# 4. Point-in-Time Recovery (특정 시점 복구)
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier synapse-prod \
  --target-db-instance-identifier synapse-prod-pitr \
  --restore-time "2026-05-07T10:00:00Z"
```

### 5.3 복구 훈련

- **분기 1회** 복구 훈련 실시
- Staging 환경에서 백업 복구 검증
- RTO(복구 시간 목표): 30분 이내
- RPO(복구 시점 목표): 5분 이내 (Point-in-Time Recovery)

## 8. 커뮤니티 서비스 운영

### 8.1 커뮤니티 모더레이션 절차

1. **신고 접수**: 관리자 대시보드 → 신고 관리 → 미처리 목록
2. **신고 검토**: 신고 상세 → 신고 내용 + 대상 콘텐츠 확인
3. **조치 결정**:
   - `dismiss`: 부당 신고. 조치 없음.
   - `warn`: 콘텐츠 작성자에게 경고 알림 발송
   - `remove`: 콘텐츠 삭제 (status → removed)
   - `suspend`: 사용자 계정 일시 정지 (반복 위반)
4. **처리 완료**: 신고자에게 결과 알림 자동 발송
5. **에스컬레이션**: 법적 이슈 → 법무팀 전달

### 8.2 스터디 그룹 관리

- **그룹 정지**: 관리자 → 스터디 그룹 관리 → 정지 버튼 → 사유 입력
- **그룹 재활성화**: 관리자 → 그룹 상세 → 활성화 버튼

### 8.3 알림 서비스 모니터링

| 지표 | 경보 기준 | 대응 |
|------|-----------|------|
| FCM 발송 성공률 | < 95% | 토큰 만료율 확인, FCM 서버 상태 점검 |
| 인앱 알림 적재 지연 | > 5초 | Kafka consumer lag 확인 |
| 이메일 발송 실패율 | > 5% | SES 쿼터/반송률 확인 |
