## B.2 네트워크 영역 모델 (Trust Zone)

Wiki 03.1 다이어그램의 영역 구분을 Zero Trust 신뢰 수준으로 재해석:

```
┌─────────────────────────────────────────────────────────┐
│ Zone 0: Internet                                        │ Trust: 0
│  - 누구나 접근 가능                                       │
└────────────────────────┬────────────────────────────────┘
                         │ TLS 1.3
                         ▼
┌─────────────────────────────────────────────────────────┐
│ Zone 1: Cloudflare Edge                                 │ Trust: 1
│  - WAF, DDoS, Bot Mgmt                                  │
│  - Origin은 Authenticated Origin Pull로 제한             │
└────────────────────────┬────────────────────────────────┘
                         │ mTLS (Cloudflare Origin Cert)
                         ▼
┌─────────────────────────────────────────────────────────┐
│ Zone 2: API Gateway (Spring Cloud Gateway 5)            │ Trust: 2
│  - JWT 검증, tenant_id 추출                              │
│  - Rate Limit (Redis Token Bucket)                      │
│  - 외부 노출 가능한 유일한 진입점                          │
└────────────────────────┬────────────────────────────────┘
                         │ mTLS (Istio Sidecar)
                         ▼
┌─────────────────────────────────────────────────────────┐
│ Zone 3: Application Services (10개 마이크로서비스)        │ Trust: 3
│  - 인증된 호출자만 접근 (JWT 또는 SPIFFE ID)              │
│  - /api/** 외부 노출용 / /internal/** 내부 전용           │
└────────────────────────┬────────────────────────────────┘
                         │ TLS + Auth (DB Credential)
                         ▼
┌─────────────────────────────────────────────────────────┐
│ Zone 4: Data Layer                                      │ Trust: 4 (최상위)
│  - PostgreSQL RLS, Redis ACL, Kafka SASL/SCRAM          │
│  - VPC 내부에서만 접근 가능                               │
└─────────────────────────────────────────────────────────┘
```

### B.2.1 Zone 간 트래픽 정책 매트릭스

| From\To | Z0 | Z1 | Z2 | Z3 | Z4 |
|---------|----|----|----|----|----|
| Z0 Internet | - | ✅ (HTTPS 443) | ❌ | ❌ | ❌ |
| Z1 Cloudflare | ❌ | - | ✅ (Origin Pull) | ❌ | ❌ |
| Z2 Gateway | ✅ (외부 IdP only) | ❌ | - | ✅ (Istio mTLS) | ❌ |
| Z3 Service | ✅ (Stripe/OpenAI 등) | ❌ | ❌ | ✅ (Istio mTLS) | ✅ (DB Auth) |
| Z4 Data | ❌ | ❌ | ❌ | ❌ | - |

## B.3 Kubernetes NetworkPolicy

### B.3.1 기본 정책: Deny-All

모든 네임스페이스에 우선 적용. 명시적으로 허용한 트래픽만 통과.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: synapse-prod
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
```

### B.3.2 Gateway 인입 정책

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: gateway-ingress
  namespace: synapse-prod
spec:
  podSelector:
    matchLabels:
      app: api-gateway
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              name: ingress-nginx
          podSelector:
            matchLabels:
              app: nginx-ingress
      ports:
        - protocol: TCP
          port: 8080
```

### B.3.3 서비스별 인입 정책 (외부 노출 서비스)

```yaml

  name: note-service-ingress

    matchLabels:
```

```yaml

        - podSelector:
            matchLabels:

        - protocol: TCP
          port: 5432                          # PostgreSQL
        - protocol: TCP
          port: 6379                          # Redis
        - protocol: TCP
          port: 9092                          # Kafka
        - protocol: TCP
          port: 9200                          # Elasticsearch
```

### B.4.5 mTLS 인증서 회전

Istio Citadel(istiod)이 자동 관리:

| 항목 | 설정 |
|------|------|
| 워크로드 인증서 TTL | **24시간** |
| 회전 시점 | TTL의 50% 경과 시 |
| Root CA TTL | 10년 (외부 PKI 연동 시 별도) |
| 인증서 발급 방식 | SPIFFE SVID (X.509) |

```yaml
# IstioOperator (cluster-level)
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  values:
    pilot:
      env:
        DEFAULT_WORKLOAD_CERT_TTL: 24h
        MAX_WORKLOAD_CERT_TTL: 48h
```

**Root CA 회전 (Phase 2)**: cert-manager + AWS Private CA 연동으로 외부 PKI 기반 신뢰 체인 구축. MVP는 Istio 자체 self-signed CA로 충분.

## B.5 `/internal/**` 경로 차단

Wiki 03.4에서 명시한 "Gateway를 거치지 않는" 내부 API의 외부 노출을 다층으로 차단.

### B.5.1 L1: Gateway 라우팅 배제

```yaml
# Spring Cloud Gateway 설정
spring:
  cloud:
    gateway:
      routes:
        - id: note-public
          uri: http://note-service:8080
          predicates:
            - Path=/api/v1/notes/**           # /api/** 만 라우팅
          filters:
            - StripPrefix=0
        # /internal/** 경로는 라우팅 규칙 자체가 없음 → 404
```

### B.5.2 L2: 서비스 측 Filter

Gateway 우회 공격에 대비해 서비스 자체에서도 `/internal/**` 호출은 mTLS 클라이언트 인증 필수.

```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class InternalPathGuardFilter implements WebFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String path = exchange.getRequest().getPath().value();
        
        if (!path.startsWith("/internal/")) {
            return chain.filter(exchange);
        }
        
        // X-Forwarded-Client-Cert 헤더 검사 (Istio mTLS 통과 시 주입)
        String clientCert = exchange.getRequest()
            .getHeaders().getFirst("X-Forwarded-Client-Cert");
        
        if (clientCert == null || !isInternalCaller(clientCert)) {
            exchange.getResponse().setStatusCode(HttpStatus.NOT_FOUND);
            return exchange.getResponse().setComplete();
        }
        
        return chain.filter(exchange);
    }
    
    private boolean isInternalCaller(String clientCert) {
        // SPIFFE ID 추출 후 화이트리스트 확인
        return clientCert.contains("URI=spiffe://cluster.local/ns/synapse-prod/sa/");
    }
}
```

## B.6 외부 IdP 호출 (Auth Service)

Wiki 03.2.4 Auth Service의 OAuth 연동을 안전하게 구현하는 표준.

### B.6.1 PKCE 강제

```java
// Authorization Code Flow with PKCE (RFC 7636)
@PostMapping("/oauth/authorize/{provider}")
public AuthorizeResponse authorize(@PathVariable String provider) {
    String codeVerifier = generateCodeVerifier();              // 43~128 chars
    String codeChallenge = sha256(codeVerifier);
    String state = generateState();                            // CSRF 방지
    
    // Redis에 짧은 TTL로 저장 (state → codeVerifier)
    redis.opsForValue().set(
        "oauth:pkce:" + state,
        codeVerifier,
        Duration.ofMinutes(10)
    );
    
    return AuthorizeResponse.builder()
        .url(buildAuthorizeUrl(provider, state, codeChallenge))
        .build();
}
```

### B.6.2 ID Token 검증

```java
public OidcUser verifyIdToken(String idToken, String provider) {
    // 1. JWK Set 캐시 조회 (Caffeine, TTL 1h)
    PublicKey publicKey = jwkCache.get(provider);
    
    // 2. 서명 검증
    Claims claims = Jwts.parserBuilder()
        .setSigningKey(publicKey)
        .build()
        .parseClaimsJws(idToken)
        .getBody();
    
    // 3. iss/aud/exp 검증
    if (!claims.getIssuer().equals(EXPECTED_ISSUERS.get(provider))) {
        throw new InvalidTokenException("Invalid issuer");
    }
    if (!claims.getAudience().equals(CLIENT_IDS.get(provider))) {
        throw new InvalidTokenException("Invalid audience");
    }
    if (claims.getExpiration().before(new Date())) {
        throw new InvalidTokenException("Token expired");
    }
    
    return OidcUser.from(claims);
}
```

### B.6.3 Refresh Token 회전 + Reuse Detection

Wiki 03.2.4의 "Refresh (7일) httpOnly Cookie" 결정에 따른 회전 정책:

```java
@Transactional
public TokenPair refresh(String refreshToken) {
    RefreshTokenRecord record = refreshTokenRepo.findByToken(hash(refreshToken))
        .orElseThrow(() -> new InvalidTokenException("Invalid refresh token"));
    
    if (record.isRevoked()) {
        // Reuse detection: 이미 사용된 토큰이 또 들어옴 → 전체 세션 무효화
        log.warn("Refresh token reuse detected: userId={}", record.getUserId());
        refreshTokenRepo.revokeAllByUserId(record.getUserId());
        auditService.recordSuspiciousActivity(record.getUserId(), "REFRESH_TOKEN_REUSE");
        throw new InvalidTokenException("Token reuse detected");
    }
    
    record.setRevoked(true);                  // 사용한 토큰은 즉시 폐기
    
    String newRefreshToken = generateRefreshToken();
    refreshTokenRepo.save(new RefreshTokenRecord(
        hash(newRefreshToken),
        record.getUserId(),
        Instant.now().plus(Duration.ofDays(7))
    ));
    
    String newAccessToken = issueAccessToken(record.getUserId());
    return new TokenPair(newAccessToken, newRefreshToken);
}
```

```yaml
      remoteRef:
        key: synapse/prod/stripe
        property: webhook_secret
```

### B.7.3 시크릿 카테고리별 회전 주기

| 카테고리 | 회전 주기 | 도구 |
|---------|----------|------|
| DB 비밀번호 | 30일 | AWS Secrets Manager 자동 회전 + Lambda |
| Stripe Secret Key | 수동 (사고 시) | Stripe Dashboard |
| OpenAI API Key | 90일 | 수동 + 알람 |
| JWT 서명 키 (HMAC) | 180일 | 회전 시 dual-key 검증 기간 7일 |
| FCM 서버 키 | 사고 시 | Firebase Console |
| OAuth Client Secret | 365일 | 각 Provider |
| Istio Root CA | 10년 | istioctl 또는 cert-manager |

### B.7.4 시크릿 사용 원칙

- **NEVER**: 환경변수에 평문 시크릿 직접 주입 (Deployment YAML에 평문 작성)
- **NEVER**: Git 레포에 `.env` 파일 커밋 (pre-commit hook으로 검사 — `gitleaks` or `detect-secrets`)
- **NEVER**: 로그에 시크릿 출력 (Logback 마스킹 패턴 적용)
- **NEVER**: 컨테이너 이미지에 시크릿 빌드 (`docker history`로 노출됨)
- **ALWAYS**: `valueFrom.secretKeyRef`로 K8s Secret 참조
- **ALWAYS**: 시크릿 접근 IAM 권한 최소화 (서비스별 별도 정책)
- **ALWAYS**: 시크릿 조회 시 CloudTrail 감사 로그 기록

### B.8.2 감사 로그 스키마

```sql
CREATE TABLE audit_logs (
    id              BIGSERIAL PRIMARY KEY,
    event_id        UUID NOT NULL UNIQUE,                 -- CloudEvents id
    tenant_id       UUID NOT NULL,
    user_id         UUID,                                  -- 시스템 이벤트는 NULL
    actor_type      VARCHAR(20) NOT NULL,                 -- USER, SYSTEM, API_KEY, ADMIN
    event_category  VARCHAR(32) NOT NULL,                 -- auth, authz, billing, ...
    event_type      VARCHAR(64) NOT NULL,                 -- auth.login.success
    severity        VARCHAR(10) NOT NULL,                 -- INFO, WARN, CRITICAL
    resource_type   VARCHAR(32),                          -- note, card, tenant, ...
    resource_id     VARCHAR(64),
    ip_address      INET,
    user_agent      TEXT,
    request_id      VARCHAR(64),
    trace_id        VARCHAR(64),
    details         JSONB,                                -- 추가 컨텍스트
    occurred_at     TIMESTAMPTZ NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    INDEX idx_tenant_time (tenant_id, occurred_at DESC),
    INDEX idx_user_time (user_id, occurred_at DESC),
    INDEX idx_event_type (event_type, occurred_at DESC)
);
```

### B.8.3 보존 정책

| 카테고리 | 보존 기간 | 저장소 |
|---------|----------|--------|
| CRITICAL (인증 실패, 권한 차단, 결제 실패) | **5년** | PostgreSQL → S3 Glacier |
| WARN (의심스러운 활동) | **2년** | PostgreSQL → S3 |
| INFO (일반 활동) | **90일** | PostgreSQL |

90일 이전 데이터는 S3로 이관 후 PostgreSQL에서 삭제. Glacier 이관은 Lifecycle Policy로 자동화.

### B.8.4 감사 로그 무결성

- audit_logs는 **append-only**: UPDATE/DELETE 권한 미부여 (DB Role 분리)
- 매일 자정에 전일 로그의 SHA-256 해시 체인 생성 → S3에 저장
- 외부 SIEM(예: AWS Security Hub) 연동 (Phase 2)

## B.10 데이터 암호화

Wiki 03.7의 "전구간 TLS + 민감 데이터 AES-256-GCM" 구현 디테일.

### B.10.1 전송 중 암호화 (Transit)

| 구간 | 방식 |
|------|------|
| Client ↔ Cloudflare | TLS 1.3 (Cloudflare 관리) |
| Cloudflare ↔ Gateway | TLS 1.3 + Cloudflare Authenticated Origin Pull |
| Gateway ↔ Service | mTLS (Istio) |
| Service ↔ Service | mTLS (Istio) |
| Service ↔ Data | TLS (PostgreSQL/Redis/Kafka SSL 활성화) |

### B.10.2 저장 시 암호화 (At Rest)

| 데이터 | 방식 | 키 관리 |
|--------|------|---------|
| RDS PostgreSQL | AWS KMS (AES-256) | AWS Customer Managed Key |
| ElastiCache Redis | AWS KMS 전송+저장 암호화 | AWS Customer Managed Key |
| S3 첨부파일 | SSE-KMS | AWS Customer Managed Key |
| EBS 볼륨 | AWS KMS | 기본 EBS 키 |
| Backup (RDS Snapshot) | KMS 자동 | 동일 키 |

### B.10.3 애플리케이션 레벨 암호화 (민감 필드)

특정 컬럼에 추가 암호화 적용 — DB 관리자도 평문 조회 불가:

```sql
-- 예: users.mfa_secret (TOTP 시크릿)
-- AES-256-GCM 암호화, 키는 AWS KMS Envelope Encryption
ALTER TABLE users 
  ADD COLUMN mfa_secret_encrypted BYTEA,        -- 암호화된 값
  ADD COLUMN mfa_secret_dek_encrypted BYTEA;    -- 암호화된 DEK
```

```java
// Envelope Encryption 패턴
public class EnvelopeEncryptionService {
    
    public EncryptedField encrypt(String plaintext) {
        // 1. AWS KMS에서 DEK(Data Encryption Key) 생성
        GenerateDataKeyResult dekResult = kmsClient.generateDataKey(
            GenerateDataKeyRequest.builder()
                .keyId(MASTER_KEY_ID)
                .keySpec(DataKeySpec.AES_256)
                .build()
        );
        
        // 2. 평문 DEK로 데이터 암호화 (AES-256-GCM)
        byte[] ciphertext = aesGcmEncrypt(plaintext, dekResult.plaintext());
        
        // 3. 평문 DEK 즉시 폐기 (메모리에서 zero out)
        Arrays.fill(dekResult.plaintext().array(), (byte) 0);
        
        // 4. 암호화된 DEK는 데이터와 함께 저장
        return new EncryptedField(ciphertext, dekResult.ciphertextBlob());
    }
}
```

대상 필드 (필수):
- `users.mfa_secret`
- `users.recovery_codes`
- `tenants.api_keys`
- `notification_preferences.email` (선택, 컴플라이언스 요구 시)
