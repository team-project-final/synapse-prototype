# 12. 코드 리뷰 규칙

> **프로젝트명**: Synapse — 통합 학습-지식 그래프 SaaS
> **버전**: v1.0
> **작성일**: 2026-05-07
> **기술 스택**: Spring Boot 4, Flutter 3.x, FastAPI, PostgreSQL 16, Redis, Elasticsearch, Kafka, K8s

---

## 1. 역할 정의

| 역할 | 책임 | 비고 |
|------|------|------|
| **Author** | PR 작성, 셀프 리뷰, 리뷰어 지정, 피드백 반영 | 코드 변경의 최종 책임자 |
| **Reviewer** | 코드 검토, 피드백 작성, Approve/Request Changes | 최소 1명 필수 |
| **Maintainer** | 최종 머지 권한, 아키텍처 결정, 분쟁 조정 | 프로젝트 오너 |

### 리뷰어 지정 기준

- 변경 도메인의 담당자 우선 배정
- 1인 개발 시: AI 도구(SonarQube) + 셀프 리뷰로 대체
- 외주 참여 시: Maintainer가 반드시 리뷰

## 2. 리뷰 프로세스

```
┌─────────────────────────────────────────────────────────────┐
│                     코드 리뷰 플로우                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [1] Author: PR 생성                                        │
│       │                                                     │
│       ├── 셀프 리뷰 완료 체크                                │
│       ├── CI 파이프라인 통과 확인                             │
│       └── 리뷰어 지정                                       │
│       │                                                     │
│  [2] Auto Review: SonarQube + Snyk 자동 분석                 │
│       │                                                     │
│  [3] Reviewer: 코드 리뷰                                    │
│       │                                                     │
│       ├── Approve → [5]                                     │
│       └── Request Changes → [4]                             │
│       │                                                     │
│  [4] Author: 피드백 반영 + Re-request Review                 │
│       │                                                     │
│       └── → [3] 재검토                                      │
│       │                                                     │
│  [5] Maintainer: 최종 확인 + Merge                           │
│       │                                                     │
│       └── Squash and Merge → main                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 프로세스 상세

| 단계 | 담당 | 행동 | 시간 제한 |
|------|------|------|-----------|
| PR 생성 | Author | 템플릿 작성, 셀프 리뷰 | - |
| 자동 분석 | CI | SonarQube, Snyk, Coverage | 10분 이내 |
| 첫 리뷰 | Reviewer | 코드 검토 및 피드백 | **24시간 이내** |
| 피드백 반영 | Author | 수정 또는 토론 | 48시간 이내 |
| 재검토 | Reviewer | 수정 확인 | 12시간 이내 |
| 머지 | Maintainer | 최종 승인 + 머지 | 즉시 |

## 3. 리뷰 체크리스트

### 3.1 기능 정확성 (Functionality)

- [ ] 요구사항(FR)을 정확히 구현했는가?
- [ ] 엣지 케이스를 처리했는가? (null, empty, boundary)
- [ ] 에러 처리가 적절한가? (예외 타입, 에러 메시지)
- [ ] 비즈니스 로직이 도메인 레이어에 위치하는가?
- [ ] API 응답 형식이 규격과 일치하는가?

### 3.2 보안 (Security)

- [ ] SQL Injection 가능성이 없는가? (파라미터 바인딩 확인)
- [ ] XSS 취약점이 없는가? (사용자 입력 이스케이프)
- [ ] 인증/인가가 올바르게 적용되었는가?
- [ ] 민감 정보가 로그에 노출되지 않는가?
- [ ] Rate Limiting이 적용되었는가? (공개 API)
- [ ] CORS 설정이 적절한가?

### 3.3 테넌트 격리 (Tenant Isolation)

- [ ] DB 쿼리에 tenant_id 필터가 적용되었는가?
- [ ] RLS 정책이 새 테이블에 설정되었는가?
- [ ] 캐시 키에 tenant_id가 포함되었는가?
- [ ] Elasticsearch 쿼리에 tenant 필터가 있는가?
- [ ] Kafka 메시지에 tenant_id가 포함되었는가?
- [ ] 파일 업로드 경로에 tenant 격리가 되어 있는가?

### 3.4 성능 (Performance)

- [ ] N+1 쿼리 문제가 없는가?
- [ ] 불필요한 데이터 로딩이 없는가? (Lazy/Eager 확인)
- [ ] 인덱스가 필요한 쿼리인가?
- [ ] 캐싱이 적절히 적용되었는가?
- [ ] 대량 데이터 처리 시 페이징을 사용하는가?
- [ ] 비동기 처리가 필요한 작업인가?

### 3.5 가독성 (Readability)

- [ ] 변수/함수/클래스 이름이 의도를 명확히 표현하는가?
- [ ] 복잡한 로직에 주석이 있는가? (왜 이렇게 했는지)
- [ ] 함수 길이가 적절한가? (30줄 이하 권장)
- [ ] 중복 코드가 없는가? (DRY 원칙)
- [ ] 코드 포맷팅 규칙을 따르는가?
- [ ] TODO/FIXME에 이슈 번호가 있는가?

### 3.6 테스트 (Testing)

- [ ] 새 기능에 대한 테스트가 추가되었는가?
- [ ] 버그 수정 시 회귀 테스트가 추가되었는가?
- [ ] 테스트가 독립적으로 실행 가능한가?
- [ ] 테스트 이름이 검증 내용을 설명하는가?
- [ ] Mock 사용이 적절한가? (과도한 mocking 지양)

---

### 3.7 커뮤니티 콘텐츠 보안 (Community Content Security)

- [ ] 공유 덱/노트 제목/설명에 XSS(Cross-Site Scripting, 웹 페이지에 악성 스크립트를 삽입하는 공격) 방지 적용
- [ ] 사용자 입력 (그룹 이름, 신고 설명, 평가 댓글) sanitization
- [ ] RLS 정책이 share_type별로 올바르게 적용되는지 확인
- [ ] 신고 API rate limit (10건/일) 검증
- [ ] Cross-service 내부 API (POST /internal/decks/copy) 인증 검증
- [ ] Kafka 이벤트 idempotency key 사용 확인 (processed_events)
- [ ] 알림 data_json에 민감 정보 미포함 확인
- [ ] 디바이스 토큰은 DB에서만 관리, 로그에 노출 금지

## 4. 피드백 작성 규칙

### 4.1 피드백 분류 접두사

| 접두사 | 의미 | 반영 필수 여부 |
|--------|------|---------------|
| `[MUST]` | 반드시 수정 필요 (보안, 버그, 데이터 손실) | 필수 |
| `[SHOULD]` | 강력 권장 (설계, 성능, 가독성) | 권장 |
| `[COULD]` | 제안 (더 나은 방법, 스타일) | 선택 |
| `[QUESTION]` | 질문 (이해를 위한 확인) | 답변 필수 |
| `[PRAISE]` | 칭찬 (좋은 코드, 배울 점) | - |
| `[NIT]` | 사소한 스타일 (컨벤션, 오타) | 선택 |

### 4.2 피드백 예시

```markdown
[MUST] 보안: RLS 필터 누락
이 쿼리에 tenant_id 조건이 빠져 있어 다른 테넌트의 데이터가 노출됩니다.

제안:
```java
@Query("SELECT n FROM Note n WHERE n.tenantId = :tenantId AND n.id = :id")
```

---

[SHOULD] 성능: N+1 문제
카드 목록 조회 시 각 카드의 덱 정보를 개별 쿼리로 가져오고 있습니다.
`@EntityGraph`나 `JOIN FETCH`로 한 번에 가져오는 것을 권장합니다.

---

[PRAISE] 깔끔한 설계
SM-2 알고리즘을 순수 함수로 분리한 것이 테스트하기 좋고 읽기도 좋습니다! 👍
```

### 4.3 금지 사항

- 인신공격, 비꼬는 표현 금지
- "왜 이렇게 했어?" 대신 "[QUESTION] 이 접근 방식을 선택한 이유가 궁금합니다"
- 대안 없는 비판 금지 — 문제점 + 제안을 함께 제시
- 코드 스타일만으로 Request Changes 금지 (린터에 위임)

## 6. 에티켓 규칙

### 6.1 Author의 에티켓

1. **PR 제출 전 셀프 리뷰를 반드시 수행한다**
   - diff를 한 줄씩 다시 읽기
   - 디버그 코드/console.log 제거 확인
   - 불필요한 파일 변경 제거

2. **PR은 작게 유지한다**
   - 하나의 PR = 하나의 목적
   - 400줄 초과 시 분할 고려
   - 리팩토링과 기능 변경을 분리

3. **컨텍스트를 충분히 제공한다**
   - 변경 이유 설명
   - 테스트 방법 안내
   - 스크린샷 첨부 (UI 변경)

4. **피드백에 방어적으로 반응하지 않는다**
   - 동의하면 수정하고 감사 표현
   - 동의하지 않으면 근거와 함께 토론
   - 모든 코멘트에 응답 (Resolve 또는 답변)

### 6.2 Reviewer의 에티켓

1. **존중하는 태도로 피드백한다**
   - "코드"를 리뷰하지, "사람"을 판단하지 않는다
   - "You" 대신 "We" 또는 수동태 사용
   - 좋은 코드에는 칭찬을 남긴다

2. **건설적인 피드백을 작성한다**
   - 문제점 + 이유 + 대안을 함께 제시
   - 코드 예시를 포함하면 이해하기 쉬움
   - 주관적 선호와 객관적 문제를 구분

3. **블로킹을 최소화한다**
   - 사소한 이슈로 전체를 블록하지 않는다
   - NIT/COULD는 Approve와 함께 남길 수 있다
   - 완벽보다 "충분히 좋은" 코드를 지향

## 7. 자동 리뷰 도구

### 7.1 SonarQube

| 항목 | 게이트 조건 |
|------|------------|
| Bugs | 0 (New Issues) |
| Vulnerabilities | 0 (New Issues) |
| Code Smells | A Rating 유지 |
| Coverage | 80% 이상 |
| Duplications | 3% 이하 |
| Security Hotspots | 전수 검토 |

### 7.2 Snyk (보안 의존성)

| 항목 | 정책 |
|------|------|
| Critical | PR 블록 — 즉시 수정 |
| High | PR 블록 — 수정 후 머지 |
| Medium | 경고 — 7일 이내 수정 |
| Low | 정보 — 백로그 등록 |

### 7.3 자동 리뷰 봇 설정

```yaml
# .github/workflows/auto-review.yml
- name: SonarQube Analysis
  uses: sonarqube-quality-gate-action@v1
  with:
    sonar-project-key: synapse
    sonar-host-url: ${{ secrets.SONAR_HOST }}
    sonar-token: ${{ secrets.SONAR_TOKEN }}

- name: Snyk Security Scan
  uses: snyk/actions/gradle@master
  with:
    args: --severity-threshold=high

- name: Coverage Comment
  uses: codecov/codecov-action@v4
  with:
    fail_ci_if_error: true
    threshold: 80%
```

### 7.4 추가 자동화

| 도구 | 역할 |
|------|------|
| **ktlint** | Kotlin 코드 스타일 자동 검사 |
| **dartanalyzer** | Flutter 정적 분석 |
| **ruff** | Python 린팅 + 포맷팅 |
| **Danger** | PR 규칙 자동 검증 (크기, 라벨, 설명) |
| **Dependabot** | 의존성 업데이트 자동 PR |

## 8. 특수 상황 가이드

### 8.1 1인 개발 시 리뷰 전략

| 단계 | 방법 |
|------|------|
| 작성 직후 | 15분 휴식 후 셀프 리뷰 |
| 자동 분석 | SonarQube + Snyk 결과 확인 |
| 시간차 리뷰 | 다음 날 다시 한번 검토 |
| 주간 회고 | 주 1회 전체 코드 되돌아보기 |

### 8.2 외주 협업 시

- 모든 PR에 Maintainer(본인) 리뷰 필수
- 보안/테넌트 격리 체크리스트 강화 적용
- 민감 정보(API 키, 인프라 설정) 변경 시 직접 처리

### 8.3 Hotfix (긴급 수정)

- 리뷰 시간 단축: 4시간 이내
- 최소 검증: 보안 + 기능 정확성
- 머지 후 리뷰 허용 (단, 24시간 내 사후 리뷰 필수)
- 사후 리뷰에서 발견된 이슈는 후속 PR로 수정
