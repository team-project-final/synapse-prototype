## C.2 CloudEvents 1.0 스키마 표준

Wiki 03.4의 결정에 따라 모든 이벤트는 [CloudEvents 1.0.2](https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/spec.md) 호환.

### C.2.1 필수 / 선택 필드

| 필드 | 타입 | 필수 | 표준/확장 | Synapse 사용법 |
|------|------|------|-----------|---------------|
| `specversion` | String | ✅ | 표준 | 항상 `"1.0"` |
| `id` | String | ✅ | 표준 | UUID v7 (시간 순서 보장) |
| `source` | URI-Reference | ✅ | 표준 | `synapse/{service-name}` (예: `synapse/note-service`) |
| `type` | String | ✅ | 표준 | `{aggregate}.{action}` (예: `note.created`) — Wiki 03.4 토픽명과 일치 |
| `time` | Timestamp | ✅ | 표준 (선택이나 필수화) | RFC 3339, UTC, microsecond |
| `subject` | String | ✅ | 표준 (선택이나 필수화) | `{resource-type}/{resource-id}` (예: `notes/uuid`) |
| `datacontenttype` | String | ✅ | 표준 | 항상 `"application/json"` |
| `data` | Object | ✅ | 표준 | 도메인별 페이로드 |
| `tenantid` | String | ✅ | **Synapse 확장** | UUID (멀티테넌시 필수) |
| `correlationid` | String | 선택 | **Synapse 확장** | 비즈니스 흐름 묶음 키 |
| `causationid` | String | 선택 | **Synapse 확장** | 이 이벤트를 유발한 이전 이벤트의 `id` |
| `userid` | String | 선택 | **Synapse 확장** | 사용자 행위 이벤트의 actor |
| `dataschema` | URI | 선택 | 표준 | 페이로드 스키마 URL (Schema Registry) |

### C.2.3 Kafka 메시지 매핑

CloudEvents Spec [Kafka Protocol Binding](https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/bindings/kafka-protocol-binding.md) 의 **Binary Mode** 사용.

```
# Kafka Headers (Binary Mode)
ce_specversion:       "1.0"
ce_id:                "01956d8a-..."
ce_source:            "synapse/card-service"
ce_type:              "card.reviewed"
ce_subject:           "cards/01956d8b-..."
ce_time:              "2026-05-18T10:30:00.123456Z"
ce_datacontenttype:   "application/json"
ce_dataschema:        "https://schemas.synapse.app/card.reviewed/v1"
ce_tenantid:          "01956d8c-..."
ce_userid:            "01956d8d-..."
ce_correlationid:     "01956d8e-..."
ce_causationid:       "01956d8f-..."
traceparent:          "00-..."                             # W3C Trace Context
tracestate:           "..."

# Kafka Message Key
{tenantid}:{subject}                                       # 파티션 라우팅

# Kafka Message Value
{
  "cardId": "...",
  "rating": 4,
  ...
}
```

**Structured Mode는 사용하지 않음** — Binary Mode가 헤더로 메타데이터를 노출하여 컨슈머가 페이로드 파싱 없이 라우팅/필터링 가능.

### C.2.4 Kafka 메시지 키 정책

| 토픽 | Key | 이유 |
|------|-----|------|
| `note.*` | `{tenantid}:notes:{noteId}` | 동일 노트 이벤트는 같은 파티션 (순서 보장) |
| `card.*` | `{tenantid}:cards:{cardId}` | 동일 카드 이벤트 순서 |
| `card.review.due` | `{tenantid}:{userId}` | 사용자별 묶음 |
| `user.*` | `{tenantid}:users:{userId}` | 사용자 라이프사이클 순서 |
| `billing.*` | `{tenantid}:subscriptions:{subId}` | 구독 변경 순서 |
| `gamification.*` | `{tenantid}:users:{userId}` | 사용자별 XP/배지 순서 |
| `audit.event` | `{tenantid}` | 테넌트별 균등 분산 |
| `community.*` | `{tenantid}:{groupId}` or `{tenantid}` | 그룹별 또는 테넌트별 |

## C.3 Schema Registry 도입

Wiki 03번에는 명시되지 않은 운영 결정. **JSON Schema 기반** Schema Registry 채택.

### C.3.1 도구 선택: Apicurio Registry

| 후보 | 장점 | 단점 | 채택 |
|------|------|------|------|
| Confluent Schema Registry | 가장 널리 사용, 안정적 | Confluent Community License (상용 제약) | ❌ |
| **Apicurio Registry** | Apache 2.0, JSON Schema/Avro/Protobuf 모두 지원, CloudEvents 통합 | 운영 사례 적음 | ✅ |
| AWS Glue Schema Registry | AWS 통합 좋음 | JSON Schema 미지원 (Avro/Protobuf만) | ❌ |

### C.3.2 Subject 네이밍 규칙

```
{topic-name}-value                                         # 페이로드 (data 필드)

예:
note.created-value
card.reviewed-value
gamification.xp.earned-value
```

CloudEvents envelope 자체는 고정 스펙이므로 Registry에 등록하지 않음 — `data` 필드의 페이로드만 등록.

### C.3.3 호환성 정책

Subject별 호환성 모드:

| 카테고리 | 호환성 | 이유 |
|---------|--------|------|
| 비즈니스 핵심 (note.*, card.*, user.*) | **FULL_TRANSITIVE** | 모든 이전 버전과 양방향 호환 |
| 분석/통계 (audit.event, gamification.*) | **BACKWARD_TRANSITIVE** | 새 컨슈머가 옛 데이터 읽을 수 있어야 함 |
| 알림 (notification.send) | **BACKWARD** | 마지막 1개 버전만 호환 (즉시 마이그레이션) |

### C.3.4 Schema Registry 클라이언트 설정

```yaml
# application.yml
synapse:
  schema-registry:
    url: http://apicurio-registry.synapse-prod:8080
    cache:
      max-size: 1000
      ttl: 5m
    auth:
      type: basic
      username: ${SCHEMA_REGISTRY_USER}
      password: ${SCHEMA_REGISTRY_PASSWORD}
```

```java
@Configuration
public class SchemaRegistryConfig {
    
    @Bean
    public SchemaResolver schemaResolver() {
        return SchemaResolverBuilder.create()
            .registryUrl(props.getUrl())
            .cacheMaxSize(props.getCache().getMaxSize())
            .cacheTtl(props.getCache().getTtl())
            .build();
    }
}
```

### C.3.5 Producer 검증 흐름

```
1. 이벤트 생성
2. Schema Resolver가 `{type}-value` subject의 최신 스키마 조회 (캐시 우선)
3. 페이로드를 스키마로 검증
4. 검증 실패 → 발행 중단, Outbox에 ERROR 상태로 기록
5. 검증 성공 → CloudEvents Binary Mode로 Kafka 발행
6. 헤더에 `ce_dataschema` URL 포함
```

### C.3.6 Consumer 검증 흐름

```
1. Kafka 메시지 수신
2. `ce_dataschema` 헤더에서 스키마 URL 추출
3. Schema Resolver가 해당 URL의 스키마 조회 (캐시)
4. 페이로드를 스키마로 검증
5. 검증 실패 → DLQ 발송 (C.7 참조)
6. 검증 성공 → 비즈니스 로직 호출
```

## C.4 호환 변경 vs 비호환 변경

### C.4.1 호환 변경 (Schema 버전 마이너 업)

| 변경 유형 | BACKWARD | FORWARD | FULL |
|----------|----------|---------|------|
| 선택 필드 추가 (default 값 있음) | ✅ | ✅ | ✅ |
| 선택 필드 제거 (default 값 있음) | ✅ | ✅ | ✅ |
| enum 값 추가 (default 있음) | ✅ | ✅ | ✅ |
| 필드 description/title 변경 | ✅ | ✅ | ✅ |
| 정수 범위 확장 (`minimum: 0` → 제거) | ✅ | ❌ | ❌ |
| 정수 범위 축소 | ❌ | ✅ | ❌ |

### C.4.2 비호환 변경 (새 토픽 v2 필요)

| 변경 유형 | 영향 |
|----------|------|
| 필수 필드 추가 (default 없음) | Producer가 새 필드를 보내면 옛 Consumer는 무시. 옛 Producer가 새 필드 없이 보내면 새 Consumer는 검증 실패 |
| 필드 타입 변경 (`string` → `integer`) | 파싱 불가 |
| 필드 이름 변경 (`cardId` → `card_id`) | 별개 필드 취급 |
| enum 값 제거 | 옛 데이터 검증 실패 |
| 필수 필드 제거 | 새 컨슈머가 필드 부재로 실패 |
| 필드 의미 변경 (예: `rating` 1-5 → 0-100) | 데이터 해석 충돌 |

### C.4.3 호환 변경 예시

```json
// v1 (현재)
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "CardReviewed",
  "type": "object",
  "properties": {
    "cardId": { "type": "string", "format": "uuid" },
    "deckId": { "type": "string", "format": "uuid" },
    "rating": { "type": "integer", "minimum": 1, "maximum": 4 },
    "interval": { "type": "integer", "minimum": 0 },
    "nextReviewAt": { "type": "string", "format": "date-time" },
    "elapsedMs": { "type": "integer", "minimum": 0 }
  },
  "required": ["cardId", "deckId", "rating", "interval", "nextReviewAt"]
}

// v1.1 (BACKWARD 호환 - 선택 필드 추가)
{
  ...
  "properties": {
    ...
    "easeFactor": {                                        // 새 선택 필드
      "type": "number",
      "minimum": 1.3,
      "default": 2.5
    }
  }
  // required 변경 없음
}
```

### C.4.4 비호환 변경 시 절차 (`v1` → `v2`)

비호환 변경 발생 시 새 토픽 생성. **`{topic}.v2`** 네이밍.

예: `card.reviewed` (v1) → `card.reviewed.v2`

```
1. 새 스키마를 `card.reviewed.v2-value` subject로 등록 (compatibility: NONE)
2. 새 토픽 `card.reviewed.v2` 생성 (파티션 수 동일)
3. Producer가 dual-write 시작 (v1 + v2 동시 발행)
4. 모든 Consumer를 v2로 마이그레이션 (점진적)
5. Consumer 마이그레이션 완료 확인 (v1 lag 모니터링)
6. Producer가 v1 발행 중단
7. v1 토픽 보관 (retention 90일 후 자동 삭제)
8. Wiki 03.4의 토픽 목록 갱신
```

## C.5 Dual-write 마이그레이션 패턴

### C.5.1 Producer 코드

```java
@Service
public class CardEventPublisher {
    
    @Value("${synapse.events.card-reviewed.dual-write:false}")
    private boolean dualWrite;
    
    @Value("${synapse.events.card-reviewed.use-v2:false}")
    private boolean useV2;
    
    @Transactional
    public void publishCardReviewed(Card card, ReviewResult result) {
        CloudEvent baseEvent = buildBase(card, result);
        
        // v1 발행 (기존 컨슈머 호환)
        if (!useV2 || dualWrite) {
            CloudEvent v1Event = baseEvent.toBuilder()
                .withType("card.reviewed")
                .withDataSchema(URI.create("https://schemas.synapse.app/card.reviewed/v1"))
                .withData(buildV1Payload(card, result))    // 옛 스키마
                .build();
            outboxRepo.save(toOutboxRecord(v1Event, "card.reviewed"));
        }
        
        // v2 발행 (새 컨슈머 + 새 필드)
        if (useV2 || dualWrite) {
            CloudEvent v2Event = baseEvent.toBuilder()
                .withType("card.reviewed.v2")
                .withDataSchema(URI.create("https://schemas.synapse.app/card.reviewed/v2"))
                .withData(buildV2Payload(card, result))    // 새 스키마
                .build();
            outboxRepo.save(toOutboxRecord(v2Event, "card.reviewed.v2"));
        }
    }
}
```

### C.5.2 단계별 Feature Flag

```yaml
# Phase 0: 정상 운영 (v1만)
synapse.events.card-reviewed.dual-write: false
synapse.events.card-reviewed.use-v2: false

# Phase 1: dual-write 시작 (양쪽 모두 발행)
synapse.events.card-reviewed.dual-write: true
synapse.events.card-reviewed.use-v2: false

# Phase 2: Consumer 마이그레이션 진행 중 (dual-write 유지)
synapse.events.card-reviewed.dual-write: true
synapse.events.card-reviewed.use-v2: false

# Phase 3: 모든 Consumer가 v2로 전환 완료
synapse.events.card-reviewed.dual-write: false
synapse.events.card-reviewed.use-v2: true

# Phase 4: v1 토픽 폐기
# (Phase 3 유지, v1 토픽 자체를 삭제)
```

### C.5.3 Consumer 측 점진적 마이그레이션

```java
@Service
public class GamificationCardListener {
    
    // 기존 v1 리스너 (Phase 1~2에서 활성)
    @KafkaListener(topics = "card.reviewed", groupId = "gamification-card-cg-v1")
    public void onCardReviewedV1(ConsumerRecord<String, byte[]> record) {
        CardReviewedV1 event = deserializer.deserialize(record);
        processReview(event.toCommon());
    }
    
    // 새 v2 리스너 (Phase 1 이후 활성)
    @ConditionalOnProperty("synapse.consumers.card-reviewed-v2.enabled")
    @KafkaListener(topics = "card.reviewed.v2", groupId = "gamification-card-cg-v2")
    public void onCardReviewedV2(ConsumerRecord<String, byte[]> record) {
        CardReviewedV2 event = deserializer.deserialize(record);
        processReview(event.toCommon());                   // 동일 처리 로직
    }
}
```

⚠️ **이중 처리 방지**: 동일 비즈니스 이벤트가 v1 + v2 양쪽에서 발행되므로 멱등성 필수 (`event.id`는 다르지만 `correlationid`나 비즈니스 키로 중복 감지).

## C.6 신규 토픽 추가 절차

### C.6.1 단계

```
1. 제안 (RFC 또는 Wiki Discussion)
   - 토픽명, 발행 서비스, 컨슈머, 페이로드 스키마
2. 페이로드 JSON Schema 작성
3. 호환성 정책 결정 (FULL/BACKWARD/...)
4. Apicurio Registry에 subject 등록 (CI 자동)
5. Kafka 토픽 생성 (Terraform/GitOps)
   - 파티션 수 결정 (예상 처리량 / 컨슈머 수)
   - retention 결정 (보통 7일 ~ 90일)
6. Producer 구현 + Outbox 통합
7. Consumer 구현 + 멱등 처리
8. DLQ 토픽 자동 생성 (C.7)
9. Prometheus Lag 알람 등록
10. Wiki 03.4 문서 업데이트
```

### C.6.2 토픽 설정 표준값

```yaml
# Terraform 예시
resource "kafka_topic" "card_reviewed" {
  name               = "card.reviewed"
  replication_factor = 3
  partitions         = 12                                  # 컨슈머 인스턴스 수 × 3
  
  config = {
    "retention.ms"          = "2592000000"                 # 30일
    "cleanup.policy"        = "delete"
    "min.insync.replicas"   = "2"
    "compression.type"      = "lz4"
    "max.message.bytes"     = "1048576"                    # 1MB
  }
}
```

| 토픽 카테고리 | retention | 파티션 | 비고 |
|--------------|-----------|--------|------|
| 비즈니스 이벤트 (note/card/user) | 30일 | 12 | 재처리 가능 |
| 알림 (notification.send) | 7일 | 6 | 짧은 라이프사이클 |
| 감사 (audit.event) | 90일 | 24 | 고처리량 + 장기 보존 |
| 통계/집계 (gamification.*) | 30일 | 12 | 재계산 가능 |

## C.7 DLQ (Dead Letter Queue) 전략

### C.7.1 네이밍 + 자동 생성

각 원본 토픽마다 자동 생성:

```
{original-topic}.dlq

예:
note.created.dlq
card.reviewed.dlq
audit.event.dlq
```

### C.7.2 발송 조건

| 조건 | 처리 |
|------|------|
| 스키마 검증 실패 | 즉시 DLQ |
| Deserialization 실패 | 즉시 DLQ |
| 비즈니스 로직 예외 (재시도 가능) | Retry 후 DLQ |
| 비즈니스 로직 예외 (재시도 불가) | 즉시 DLQ |
| Poison Pill (반복 실패) | 5회 재시도 후 DLQ |

### C.7.3 Spring Kafka 설정

```java
@Configuration
public class KafkaListenerConfig {
    
    @Bean
    public DefaultErrorHandler errorHandler(KafkaTemplate<String, byte[]> kafkaTemplate) {
        DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(
            kafkaTemplate,
            (record, exception) -> new TopicPartition(record.topic() + ".dlq", record.partition())
        );
        
        // 5회 재시도, exponential backoff (1s, 2s, 4s, 8s, 16s)
        ExponentialBackOffWithMaxRetries backOff = new ExponentialBackOffWithMaxRetries(5);
        backOff.setInitialInterval(1000L);
        backOff.setMultiplier(2.0);
        backOff.setMaxInterval(60000L);
        
        DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);
        
        // 즉시 DLQ 보낼 예외
        handler.addNotRetryableExceptions(
            SchemaValidationException.class,
            DeserializationException.class,
            BusinessException.class                        // 명시적 fatal 비즈니스 에러
        );
        
        return handler;
    }
}
```

## C.8 이벤트 재처리 (Replay)

### C.8.1 재처리가 필요한 시나리오

| 시나리오 | 방식 |
|---------|------|
| 컨슈머 버그 수정 → 누락된 처리 복구 | 새 컨슈머 그룹으로 from-earliest |
| 새 통계/분석 추가 → 과거 데이터 집계 필요 | 임시 컨슈머로 retention 내 데이터 처리 |
| 데이터 손상 복구 | 보상 이벤트 발행 또는 selective replay |

### C.8.2 컨슈머 그룹 재설정

```bash
# 1. 컨슈머 중단
kubectl scale deployment gamification-service --replicas=0

# 2. 오프셋 리셋
kafka-consumer-groups --bootstrap-server kafka:9092 \
  --group gamification-card-reviewer-cg \
  --topic card.reviewed \
  --reset-offsets --to-earliest \
  --execute

# 3. 컨슈머 재시작
kubectl scale deployment gamification-service --replicas=3
```

### C.8.3 멱등성 검증 필수

C.2의 Idempotency 패턴이 정상 동작하면 replay 시 중복 처리 안 됨. 단, processed_events 테이블 retention(90일) 이전 이벤트는 재처리됨 — 의도된 동작.

### C.8.4 시간 범위 부분 replay

특정 기간만 재처리하고 싶을 때:

```bash
# 2026-05-15 00:00 ~ 23:59 사이의 메시지만 새 임시 컨슈머로 처리
kafka-consumer-groups --bootstrap-server kafka:9092 \
  --group temp-replay-20260515 \
  --topic card.reviewed \
  --reset-offsets --to-datetime 2026-05-15T00:00:00.000Z \
  --execute
```

## C.9 CI/CD 검증 게이트

### C.9.1 스키마 변경 PR 시 자동 검증

```yaml
# .github/workflows/schema-validate.yml
name: Schema Compatibility Check

on:
  pull_request:
    paths:
      - 'schemas/**/*.json'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Install apicurio-registry-cli
        run: |
          curl -L https://github.com/apicurio/apicurio-registry/releases/download/...
      
      - name: Validate schema syntax
        run: |
          for f in schemas/*.json; do
            jq -e . "$f" > /dev/null || exit 1
          done
      
      - name: Check compatibility against current
        run: |
          for f in schemas/*.json; do
            subject=$(basename "$f" .json)
            apicurio-cli test-compatibility \
              --registry-url ${{ secrets.SCHEMA_REGISTRY_URL }} \
              --subject "$subject" \
              --schema "$f" || exit 1
          done
      
      - name: Block on breaking change
        run: |
          # apicurio-cli가 호환성 위반 시 non-zero exit
          echo "All schema changes are backward compatible"
```

### C.9.2 PR 머지 시 자동 등록

```yaml
# .github/workflows/schema-register.yml
on:
  push:
    branches: [main]
    paths:
      - 'schemas/**/*.json'

jobs:
  register:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Register schemas
        run: |
          for f in schemas/*.json; do
            subject=$(basename "$f" .json)
            apicurio-cli register \
              --registry-url ${{ secrets.SCHEMA_REGISTRY_URL }} \
              --subject "$subject" \
              --schema "$f"
          done
```
