## A.2 통신 방식 선택 의사결정 트리

```
호출자가 응답을 기다려야 하는가?
│
├─ Yes → 결과가 클라이언트 응답에 포함되는가?
│   │
│   ├─ Yes → 외부 노출? 
│   │   ├─ Yes (사용자 직접) → REST via Gateway
│   │   └─ No  (서비스 간)   → ★ gRPC (internal)
│   │
│   └─ No  → 측면 효과만 필요 → ★ Kafka (비록 응답을 기다리더라도)
│
└─ No → 결과가 다른 서비스에 영향 주는가?
    │
    ├─ Yes → ★ Kafka (도메인 이벤트)
    └─ No  → 호출 자체가 불필요할 가능성 → 재검토
```

### 추가 가이드

| 상황 | 선택 | 이유 |
|------|------|------|
| AI 응답 스트리밍 (Q&A, 카드 생성) | gRPC server-streaming | 점진적 응답, 백프레셔 가능 |
| Card → Note 즉시 조회 (카드 생성 시 노트 본문 필요) | gRPC | 응답 흐름 내 동기 필요 |
| Community → Card 덱 복사 | gRPC `POST /internal/decks/copy` | Wiki 03.4에 명시된 내부 API |
| Note 작성 후 검색 인덱싱 / 임베딩 | Kafka `note.created` | 사용자 응답 무관, 비동기 |
| Card 복습 후 XP/배지 처리 | Kafka `card.reviewed` | 결과를 즉시 보여줄 필요 없음 |
| 결제 Webhook → 구독 변경 알림 | Kafka `billing.subscription.changed` → fan-out | 여러 컨슈머 필요 |

### A.3.2 Deadline 표준값

**원칙**: 모든 gRPC 호출에 deadline 필수. 기본값 사용 금지.

| 호출 유형 | Deadline | 비고 |
|----------|----------|------|
| 동일 클러스터 단순 조회 | **500ms** | Card → Note Get 등 |
| 동일 클러스터 쓰기 | **1s** | Community → Card 덱 복사 |
| AI 호출 (비스트리밍) | **15s** | 짧은 카드 생성 |
| AI 호출 (스트리밍) | **30s** (per-message: 5s) | Q&A 응답 |
| OAuth 외부 호출 | **3s** | Google/GitHub/Apple/MS |
| Stripe API 호출 | **10s** | Stripe SLA 권장값 |

### A.3.4 데코레이터 적용 순서

```
요청 → Bulkhead → TimeLimiter → CircuitBreaker → Retry → 실제 호출
```

Spring `@Bulkhead` + `@TimeLimiter` + `@CircuitBreaker` + `@Retry` 어노테이션 4개 동시 적용 시 위 순서 보장됨.

```java
@CircuitBreaker(name = "note-service", fallbackMethod = "getNoteFallback")
@Retry(name = "idempotent-grpc")
@TimeLimiter(name = "note-service")
@Bulkhead(name = "note-service", type = Bulkhead.Type.SEMAPHORE)
public CompletableFuture<NoteSummary> getNoteForCard(UUID noteId) {
    return CompletableFuture.supplyAsync(() ->
        noteServiceStub
            .withDeadlineAfter(500, TimeUnit.MILLISECONDS)
            .getNoteSummary(...)
    );
}

private CompletableFuture<NoteSummary> getNoteFallback(UUID noteId, Throwable t) {
    log.warn("Note service fallback triggered for noteId={}", noteId, t);
    return CompletableFuture.completedFuture(NoteSummary.empty(noteId));
}
```

### A.5.3 멱등 처리 패턴 (CloudEvents `id` 기반)

Wiki 03.4의 CloudEvents 스키마에 따르면 모든 이벤트는 `id` (uuid-v7) 필드를 가진다. 이를 멱등키로 사용.

```java
@Service
public class IdempotentEventProcessor {
    
    private final ProcessedEventRepository processedRepo;
    private final RedisTemplate<String, String> redis;
    
    @Transactional
    public <T extends CloudEvent> void process(T event, Consumer<T> handler) {
        String key = "evt:processed:" + event.getId();
        
        // L1: Redis SETNX (빠른 중복 감지)
        Boolean acquired = redis.opsForValue()
            .setIfAbsent(key, "1", Duration.ofDays(7));
        
        if (Boolean.FALSE.equals(acquired)) {
            log.debug("Event already processed in Redis: {}", event.getId());
            return;
        }
        
        // L2: DB UNIQUE 제약 (최종 방어선)
        if (processedRepo.existsByEventId(event.getId())) {
            log.debug("Event already in DB: {}", event.getId());
            return;
        }
        
        try {
            handler.accept(event);
            processedRepo.save(new ProcessedEvent(
                event.getId(),
                event.getType(),
                event.getTime(),
                Instant.now()
            ));
        } catch (Exception e) {
            // Redis 키 회수 (다음 재시도 허용)
            redis.delete(key);
            throw e;
        }
    }
}
```

```sql
CREATE TABLE processed_events (
    event_id      UUID PRIMARY KEY,
    event_type    VARCHAR(64) NOT NULL,
    occurred_at   TIMESTAMPTZ NOT NULL,
    processed_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_processed_at ON processed_events(processed_at);

-- 90일 이전 데이터 정리 (월 1회 배치)
DELETE FROM processed_events WHERE processed_at < NOW() - INTERVAL '90 days';
```

## A.6 Transactional Outbox 구현 표준

Wiki 03번에서 명시적으로 다루지 않지만, **Kafka 발행을 하는 모든 서비스에 필수**.

### A.6.1 스키마

```sql
CREATE TABLE outbox_event (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       UUID NOT NULL,
    aggregate_id    VARCHAR(64) NOT NULL,
    aggregate_type  VARCHAR(64) NOT NULL,
    event_type      VARCHAR(64) NOT NULL,                 -- CloudEvents `type`
    event_id        UUID NOT NULL UNIQUE,                 -- CloudEvents `id`
    payload         JSONB NOT NULL,
    trace_id        VARCHAR(64),
    span_id         VARCHAR(64),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    published_at    TIMESTAMPTZ
);

CREATE INDEX idx_outbox_unpublished 
    ON outbox_event(created_at) 
    WHERE published_at IS NULL;
```

### A.6.2 폴링 방식 Outbox Relay (MVP)

```java
@Component
public class OutboxRelay {
    
    @Scheduled(fixedDelay = 500)                          // 500ms 폴링
    @Transactional
    public void relay() {
        List<OutboxEvent> batch = outboxRepo.findUnpublished(
            PageRequest.of(0, 100, Sort.by("createdAt"))
        );
        
        for (OutboxEvent event : batch) {
            try {
                ProducerRecord<String, byte[]> record = buildRecord(event);
                injectTraceHeaders(record, event.getTraceId(), event.getSpanId());
                
                kafkaTemplate.send(record).get(2, TimeUnit.SECONDS);
                
                event.setPublishedAt(Instant.now());
                outboxRepo.save(event);
            } catch (Exception e) {
                log.error("Outbox publish failed: {}", event.getId(), e);
                // 다음 폴링 사이클에서 재시도
            }
        }
    }
}
```

### A.7.2 Kafka 헤더 전파 (수동)

Kafka Producer / Consumer는 자동 전파가 안정적이지 않으므로 명시적 전파:

```java
// Producer
public void publish(CloudEvent event) {
    Span span = tracer.spanBuilder("kafka.publish." + event.getType())
        .setAttribute("messaging.system", "kafka")
        .setAttribute("messaging.destination", topic)
        .setAttribute("synapse.event.id", event.getId())
        .startSpan();
    
    try (Scope scope = span.makeCurrent()) {
        ProducerRecord<String, byte[]> record = new ProducerRecord<>(topic, key, payload);
        
        // W3C Trace Context 헤더 주입
        W3CTraceContextPropagator.getInstance().inject(
            Context.current(),
            record.headers(),
            (carrier, k, v) -> carrier.add(k, v.getBytes(StandardCharsets.UTF_8))
        );
        
        kafkaTemplate.send(record).get();
    } finally {
        span.end();
    }
}

// Consumer
@KafkaListener(topics = "note.created")
public void onNoteCreated(ConsumerRecord<String, byte[]> record) {
    Context extracted = W3CTraceContextPropagator.getInstance().extract(
        Context.current(),
        record.headers(),
        new TextMapGetter<>() { /* implements get/keys */ }
    );
    
    try (Scope scope = extracted.makeCurrent()) {
        Span span = tracer.spanBuilder("kafka.consume.note.created")
            .setAttribute("messaging.system", "kafka")
            .startSpan();
        try (Scope s = span.makeCurrent()) {
            processEvent(record);
        } finally {
            span.end();
        }
    }
}
```

```
1. PagerDuty 알람 수신
2. Grafana 대시보드 확인
   - 어느 서비스의 어느 호출이 실패하는가?
   - Circuit Breaker가 Open 상태인가?
   - Kafka Lag이 증가하는가?
3. Jaeger에서 trace 추적
   - 가장 느린 span은 어디인가?
   - 외부 의존성(OpenAI, Stripe, OAuth)인가?
4. 임시 조치
   - Feature Flag로 해당 기능 비활성화
   - 트래픽 Throttle
   - 회로 강제 Open (수동)
5. 사후 분석
   - Post-mortem 문서 작성 (Wiki 16번 운영 메뉴얼 템플릿)
   - Resilience4j 임계치 조정 검토
```
